# Design

> *How hortus is put together, and why.*

This document is for someone (a person or a future model) who wants to
extend hortus, understand the choices, or know where to put a new command.
The README is the front gate. The essay is the long letter. This is the
blueprint.

## architecture, in one paragraph

Hortus is a single Rust binary. The directory layout is the source of
truth; the binary is a gardener that reads, manipulates, and writes
files. The runtime stack is: clap for CLI parsing, serde_yaml and toml for
file formats, pulldown-cmark for the markdown in `bloom.html`, ratatui-free
terminal output via owo-colors, rand for `wander` and `quote`, and rusqlite
*not* — the tool is database-free. The `bloom.html` graph is rendered
client-side in vanilla JavaScript, no framework, no CDN.

The codebase is small enough to read in an afternoon:

```
src/
├── main.rs          # clap CLI wiring
├── lib.rs           # re-exports cmd, model, text
├── model.rs         # Garden, Seed, Bed, Climate types + I/O
├── text.rs          # tokenize, idf, document_frequencies
└── cmd/
    ├── plant.rs     # plant, plant --interactive
    ├── sow.rs       # place a seed in a bed
    ├── tend.rs      # open in $EDITOR, mark watered
    ├── list.rs      # filtered listing
    ├── wander.rs    # random bed or stale seeds
    ├── quote.rs     # random seed
    ├── compost.rs   # archive / restore
    ├── cross.rs     # TF-IDF link discovery
    ├── merge.rs     # fuse two seeds
    ├── unmerge.rs   # split a merged seed
    ├── tag.rs       # add/remove/list tags
    ├── climate.rs   # show/set the weather
    ├── diary.rs     # weekly journal entry
    ├── letter.rs    # one-page letter to self
    ├── today.rs     # daily landing
    └── bloom.rs     # terminal mosaic + bloom.html + index.md

tests/
└── garden.rs        # end-to-end tests with a Mutex-locked HORTUS_ROOT
```

## the file format

The directory layout is the data model. There is no schema in code, no
migration, no DDL. The types in `model.rs` are a *projection* of what's on
disk.

```
my-hortus/
├── seeds/YYYY-MM-DD-<slug>.md    # one file per live thought
├── compost/YYYY-MM-DD-<slug>.md  # archived, with composted_at + epitaph
├── beds/<slug>.md                # thematic collection
├── letters/YYYY-MM-DD-letter.md  # written by `hortus letter`
├── diary/YYYY-MM-DD-diary.md     # written by `hortus diary`
├── climate.toml                  # mood / reading / season + history
├── index.md                      # regenerated by `hortus bloom`
└── bloom.html                    # regenerated by `hortus bloom`
```

A seed file is YAML frontmatter + a markdown body:

```markdown
---
id: 2026-06-29-a-rose-by-any-other-name
planted: 2026-06-29
last_tended: 2026-06-29
mood: curious
tags: [language, metaphor]
---

Naming is a small violence...
```

The frontmatter is parsed by `serde_yaml`. The body is everything after
the closing `---`. The `is_composted` field on `Seed` is *not* serialized
— it is derived at load time from the file's parent directory. This means
the file's location is the source of truth for liveness, not the
frontmatter.

A bed file is the same shape: frontmatter with a `name` and a `seeds:`
list, body with a description.

`climate.toml` is TOML. Two sections: `[now]` (the current weather) and
`[[history]]` (snapshots of past nows, taken every time the now is
updated).

## the data model

```rust
pub struct Garden { root: PathBuf }
pub struct Seed {
    id, planted, last_tended, mood, tags,
    composted_at, epitaph, body, is_composted,
}
pub struct Bed { name, seeds, description }
pub struct Climate { now: ClimateNow, history: Vec<ClimateSnapshot> }
```

`Seed` is the unit of thought. `Bed` is a thin index — just a name and a
list of seed ids. `Climate` is orthogonal to the rest of the garden. A
`Garden` is just a path; it doesn't hold data.

The split between `model.rs` (types + I/O) and `cmd/*` (verbs) is the
architectural line. `model.rs` knows nothing about clap. `cmd/*` is
allowed to do filesystem and stdout things but is encouraged to keep
business logic in helper functions that are independently testable.

## the commands

Each command is a subcommand of the main binary, declared in `main.rs` via
clap derive. Each lives in its own file under `src/cmd/`. The pattern is
the same in every file:

```rust
#[derive(Args)]
pub struct XArgs { /* clap flags */ }

pub fn run(args: XArgs) -> Result<()> {
    let garden = Garden::discover(None)?;
    // ... do the thing ...
    Ok(())
}
```

`Garden::discover` reads `HORTUS_ROOT` from the environment, or defaults
to `./my-hortus` if present, else the current directory. The `--root`
global flag sets the env var. Integration tests use a `Mutex` to serialize
HORTUS_ROOT mutations across parallel test threads.

## the bloom pipeline

`hortus bloom` is the most complex command. It produces three artifacts:

1. **Terminal mosaic** — printed to stdout. A small sparkline of
   seeds-per-day, beds, mood histogram.
2. **`bloom.html`** — a self-contained file with a force-directed graph
   rendered in vanilla JavaScript. Nodes are seeds. Edges connect seeds
   that share a bed (solid moss) or a tag (dashed rose). Composted seeds
   are faded. A side panel renders each seed's body as markdown via
   pulldown-cmark.
3. **`index.md`** — a markdown summary of the garden, with the bed list,
   climate, and the ten most recent seeds.

The HTML is generated by replacing placeholders in `src/cmd/bloom_template.html`
with computed values (`__NODES__`, `__EDGES__`, `__MOOD_DATA__`, etc.). The
template is `include_str!`'d at compile time, so it is part of the binary.
No CDN, no build step. Open the file from the filesystem and it works.

The mood timeline at the top of `bloom.html` is a per-day bar chart,
colored by the dominant mood of the day. Colors are hardcoded in
`mood_color()` — a small categorical palette that maps mood strings
("quiet", "tender", "hopeful", "curious", ...) to hex codes in the
garden palette.

## the cross-pollination algorithm

`hortus cross` is the magic command. The algorithm is plain TF-IDF:

1. Tokenize each live seed's body (lowercase, stopwords removed,
   short tokens dropped). Stopword list is in `text.rs`.
2. Compute document frequencies: for each token, how many seeds contain it.
3. For each pair of seeds that doesn't already share a bed:
   - Find shared tokens.
   - Score = Σ idf(token) for each shared token, normalized by the
     smaller seed's token count.
4. Sort pairs by score, take the top N, print.

No model, no API, no GPU. The signal is real: shared rare words are a
honest indicator of latent connection. The transparency is part of the
point.

## the letter generator

`hortus letter` is a templated letter, not a generated one. It picks
three seeds distributed across the time window (early, middle, late),
quotes the first sentence of each, and weaves a mood-aware closing. The
templates are in `letter.rs`:

```rust
let opening = match (n, reading) {
    (0, _) => "...",
    (_, Some(r)) => format!("I notice you've been {} this past while. I've been reading {}, ..."),
    (_, None) => format!("I notice you've been {} this past while. ..."),
};
```

The closing rotates by mood:

```rust
let closing = match m {
    m if m.contains("quiet") => "tend what is already growing",
    m if m.contains("restless") => "plant something small and let it lie",
    m if m.contains("tender") => "be gentle with the next thing you write",
    // ...
};
```

The intent is that the letter reads like a letter, not like a list. On a
good day it does. On a bad day it doesn't. Both are part of the metaphor.

## testing

Two test surfaces:

- **Unit tests** inline in `model.rs`, `text.rs`, `cmd/letter.rs`,
  `cmd/bloom.rs`, `cmd/list.rs`, `cmd/unmerge.rs`. Pure-function tests of
  slug, frontmatter parsing, id generation, tokenization, mood colors,
  first-sentence extraction.
- **Integration tests** in `tests/garden.rs`. Each test sets `HORTUS_ROOT`
  to a fresh temp dir, exercises the high-level commands, and asserts on
  the resulting filesystem state. A `Mutex<()>` serializes tests
  because they share the env var.

CI runs on every push: `cargo fmt --check`, `cargo clippy -D warnings`,
`cargo test`, `cargo build --release`. The badge in the README reflects
the latest `main` commit.

## extension points

If you want to add a new command, the pattern is:

1. Create `src/cmd/<name>.rs` with a `<Name>Args` struct and a `run`
   function.
2. `pub mod <name>;` in `src/cmd/mod.rs`.
3. Add a variant to the `Command` enum in `src/main.rs`, and a match arm
   in `main()`.
4. Update the README's verbs table.

If you want to add a new field to `Seed`:

1. Add it to the struct in `model.rs` with `#[serde(default)]` so old
   files keep loading.
2. Update the `Seed` literal in `cmd/plant.rs` (one place).
3. Update any other commands that construct `Seed` literals.
4. Add a test that roundtrips a seed with the new field.

If you want to change the file format, you are entering a versioning
problem. The current design has no migration path; the file format is
implicit in the code. This is fine for a personal tool. If the format
ever needs to change, the right move is to bump a version field on
`climate.toml` (the only file with a clear identity) and add a migration
in `Climate::load_or_default`. For seeds and beds, the YAML is forgiving
enough that additive changes (new optional fields) work without
migration.

## performance

- The largest example garden I have measured has ~100 seeds. Loading
  takes < 10 ms. The bloom graph simulation runs in < 1 s for that size.
- TF-IDF cross is O(n²) over seeds but the constant is small. 100 seeds
  takes < 100 ms. The O(n²) ceiling will eventually bite, but a
  threshold-based prefilter (drop pairs whose largest shared-word
  frequency exceeds some cutoff) would extend the range to thousands.
- `bloom.html` JS runs the force simulation for 600 ticks by default. For
  100 nodes that's a few seconds on a cold laptop. The `ticks` constant
  is hardcoded; a future version could expose it.

There is no profiling in CI. The tool is small enough that the
optimization work hasn't been necessary.

## what this design rules out

A few things the current architecture makes hard or impossible:

- **Concurrent access from multiple binaries.** `Seed::save` does
  read-modify-write on a file. Two processes writing at the same time
  will clobber. The `.hortus/cache` directory exists for future
  locking, but no locking is implemented.
- **Remote gardens.** There is no client/server model. The directory is
  local. If you want to sync, you sync the directory (via git, rsync,
  whatever) — the binary has no opinion.
- **A real database.** No SQLite, no LMDB, no sled. The cost is that
  everything is parsed on every command. The benefit is that the
  directory is the database, and you can `grep` it.

These are not missing features. They are consequences of the design.
Adding any of them would be a fork in the road, not an extension.
